# TASK.md — План разработки

## 1. Суть продукта

Backend-сервис для тренажёра "предскажи вывод Python-программы":
- Пользователю показывают сниппет кода на Python.
- Даются несколько вариантов ответа (multiple choice) — один правильный.
- Есть уровни сложности (3 шт.) и темы (строки, массивы/списки, циклы и т.д.).
- Клиент — Telegram Mini App (фронт не в этом репо, только backend).

Ключевой вопрос, который нужно закрыть до кодирования: это **статический банк вопросов**
(заранее подготовленные сниппеты + варианты ответов, лежащие в БД) или сервис должен
**сам генерировать/исполнять Python-код** для проверки ответа? Предполагаю первое
(проще, безопаснее, соответствует правилу "не усложнять"), исполнение чужого Python-кода
на сервере — это отдельный контур безопасности (sandboxing), которого стоит избежать,
если не подтверждено обратное.

## 2. Решения, подтверждённые пользователем

1. **Авторизация**: обязательна валидация Telegram `initData` (HMAC-проверка подписи по
   боту-токену). Анонимного доступа нет — каждый запрос идентифицирует пользователя
   по `telegram_id` из initData.
2. **Статистика**: хранится история ответов пользователя (какой вопрос, какой вариант выбрал,
   правильно/нет, когда).
3. **Наполнение банка вопросов**: только сиды — SQL-миграции/seed-скрипты. Админ-API не нужен.
4. **Формат выдачи**: "лента" вопросов без повторов — сервис должен исключать вопросы, которые
   пользователь уже видел/отвечал, при выдаче следующего вопроса. Это требует хранить связь
   пользователь → просмотренные/отвеченные вопросы (что естественно вытекает из истории ответов
   в п.2 — отдельной таблицы "просмотров" не нужно, используется таблица попыток).
5. Правильность ответа проверяется на сервере; вариант ответа в выдаче вопроса не помечается
   как правильный (утечка правильного ответа в исходном запросе недопустима).

## 3. Архитектура (упрощена по запросу пользователя)

Плоская структура из 4 слоёв, без деления на под-пакеты по доменам (`question/`, `session/` и т.п.
убраны). Один пакет на слой, каждый эндпоинт добавляет функцию/метод в соответствующий слой.

```
cmd/
  api/                # main.go — точка входа, поднимает http-сервер, сборка зависимостей
internal/
  config/             # чтение конфигурации (env)
  handler/             # HTTP-слой: роутинг, handlers, middleware (в т.ч. Telegram initData)
  service/             # бизнес-логика: формирование ленты, проверка ответа, статистика
  model/               # структуры данных (Question, Topic, User, Attempt, DTO)
  repo/                # доступ к Postgres: интерфейсы + реализация (без под-пакетов на таблицу)
migrations/           # SQL-миграции
openapi.yaml
Dockerfile
docker-compose.yaml
```

Правила:
- `handler` знает только про `service`, `service` — только про `repo` и `model`. Без обратных
  зависимостей.
- Один `Service` (или несколько простых структур в пакете `service`, без generic-абстракций)
  и один `Repo`/набор репозиториев в пакете `repo` — без интерфейсов "на будущее", если нет
  второй реализации.
- Без DI-фреймворков, без CQRS/event bus, без отдельного домена/под-пакетов — минимальная
  layered-архитектура (handler → service → repo → model).

## 4. Данные (финальная черновая модель)

- `topics` (id, name) — темы: строки, массивы, циклы и т.д.
- `questions` (id, topic_id, difficulty [1..3], code_snippet, created_at)
- `answer_options` (id, question_id, text, is_correct) — `is_correct` никогда не уходит в API-ответ
  на выдачу вопроса.
- `users` (id, telegram_id unique, created_at) — создаётся/находится по данным из initData.
- `attempts` (id, user_id, question_id, selected_option_id, is_correct, created_at) — история
  ответов; одновременно служит источником для фильтра "не повторять вопросы" (по паре
  user_id+question_id) и для статистики.

Индекс: `attempts (user_id, question_id)` — для быстрой фильтрации уже отвеченных вопросов
при формировании ленты, `(user_id, created_at)` — при необходимости отдавать историю/статистику.

## 5. API (финальный черновой набор, MVP)

Все эндпоинты, кроме `/healthz`, требуют заголовок с Telegram `initData` (например,
`Authorization: tma <initData>` по конвенции Telegram Mini Apps); middleware валидирует подпись
и резолвит/создаёт пользователя.

- `GET /questions/feed?difficulty=&topic=&limit=` — следующая порция вопросов ленты, исключая уже
  отвеченные текущим пользователем (без указания правильного варианта).
- `POST /questions/{id}/answer` — отправить выбранный вариант; сохраняет попытку в `attempts`,
  возвращает результат (правильно/неправильно + правильный вариант).
- `GET /topics` — список доступных тем.
- `GET /users/me/stats` — агрегированная статистика текущего пользователя (кол-во ответов,
  процент правильных, по темам/сложности — по необходимости).
- `GET /healthz` — healthcheck, без авторизации.

Точный контракт фиксируется в `openapi.yaml` перед реализацией.

## 6. Стек и инструменты

- Go (net/http + chi/gorilla mux — выбрать один лёгкий роутер, без тяжёлых фреймворков типа gin,
  если нет причины).
- Postgres, драйвер `pgx` или `database/sql` + `lib/pq`.
- Миграции: `golang-migrate` или `goose` (выбрать один).
- Конфигурация через переменные окружения (без Viper, если хватает `os.Getenv`).
- Тесты: стандартный `testing`, минимум unit-тесты для сервисного слоя + одна integration-проверка
  работы с БД (например, через testcontainers или docker-compose в CI — решить по месту).

## 7. Этапы реализации

1. **Уточнение требований** — выполнено, см. раздел 2.
2. **Каркас проекта** — `go.mod`, структура папок `cmd/api`, `internal/{config,handler,service,
   repo,model}`, `Dockerfile`, `docker-compose.yaml` (app + postgres).
3. **OpenAPI-контракт** — `openapi.yaml` с финальными эндпоинтами до написания кода handlers.
4. **model + миграции** — структуры `Question`, `Topic`, `User`, `Attempt`, SQL-миграции таблиц.
5. **repo** — доступ к Postgres (CRUD/выборки под нужды service), без лишних интерфейсов.
6. **service** — формирование ленты без повторов, проверка ответа и запись попытки, статистика.
7. **handler** — роутинг, HTTP-обработчики, middleware валидации Telegram initData, маппинг
   ошибок в HTTP-коды.
8. **Сиды данных** — наполнение БД примерами вопросов (строки/массивы/циклы, 3 уровня сложности)
   для проверки работоспособности.
9. **Тесты** — unit на service, минимальный smoke-тест handler-слоя.
10. **Документация** — README с инструкцией запуска через docker-compose, сверка openapi.yaml
    с реальными handlers.
11. **Финальная проверка** — прогон через docker-compose, ручная проверка эндпоинтов.

## 8. Риски / решения, которые нужно принять явно

- **Исполнение кода на сервере**: если в будущем понадобится генерировать вопросы динамически
  через реальный запуск Python — это отдельная задача с sandboxing (gVisor/Docker-in-Docker/
  ограничение ресурсов), не делать в MVP без явного запроса.
- **Утечка правильного ответа**: эндпоинт получения вопроса не должен возвращать `is_correct`
  в теле ответа — фильтровать на уровне сервиса/DTO.
- **Валидация initData**: делать её строго по официальному алгоритму Telegram (HMAC-SHA256 от
  bot-токена), хранить bot-токен только в переменных окружения, не логировать initData целиком.
- **Исчерпание ленты**: если у пользователя закончились неотвеченные вопросы в рамках
  фильтра — решить (после MVP) сбрасывать историю показов или сообщать "вопросы закончились";
  не проектировать это заранее сверх простого случая "вернуть пустой список".
